# 品証パニック（Quality Panic）— ソロモード実装指示書

## 概要

品質保証テーマのプッシュ・ユア・ラック × デッキ汚染カードゲーム。
ソロモード（1人プレイ）のデジタルプロトタイプを作成する。

- **技術スタック**: React + Vite + TypeScript
- **スタイリング**: Tailwind CSS
- **状態管理**: React useState/useReducer（外部ライブラリ不要）
- **デプロイ**: ローカル開発のみ（npm run dev で動けばOK）

---

## ゲームルール（ソロモード）

### 勝利条件
- 8ラウンド制。目標スコア到達でクリア
- 難易度別目標: 簡単15点 / 普通20点 / 難しい25点 / 品証の神30点

### カード構成（82枚）

#### 出荷山札（初期54枚）
| カテゴリ | 種類 | 枚数 | 値 |
|---|---|---|---|
| 製品 | 量産品 | 20 | 利益1点 |
| 製品 | 高付加価値品 | 10 | 利益2点 |
| 製品 | 大型案件 | 6 | 利益3点 |
| 不具合（初期） | 軽微（黄） | 6 | 不具合Pt: 1 |
| 不具合（初期） | 重大（赤） | 2 | 不具合Pt: 2 |
| イベント | SNS炎上 | 2 | 次の不具合Ptが2倍 |
| イベント | 納期プレッシャー | 2 | あと最低2枚めくらないといけない |
| イベント | ベテラン退職 | 1 | パニック閾値が一時的に2に下がる |
| イベント | 改善提案 | 2 | 不具合1枚を無効化 |
| イベント | ISO監査 | 1 | 不具合Pt=0ならボーナス3点 |
| イベント | 新人配属 | 2 | 次ラウンドのみ、1枚目を見てから続行/中止選択可 |

#### 汚染ストック（18枚、上から順に投入）
| 順番 | 種類 | 枚数 | 不具合Pt |
|---|---|---|---|
| 1〜8 | 軽微（黄） | 8 | 1 |
| 9〜14 | 重大（赤） | 6 | 2 |
| 15〜18 | リコール級（黒） | 4 | 3（即パニック） |

#### 対応カード（10枚、別山札）
| 名前 | 枚数 | 効果 | 使用後 |
|---|---|---|---|
| 応急処置 | 4 | 不具合1枚を無効化（Pt加算なし） | → 捨て山（リサイクル） |
| 原因調査 | 3 | 無効化 + 山札から不具合1枚除外 | → 捨て山（リサイクル） |
| 水際検査 | 2 | 次の不具合も自動無効化（2枚分） | → 捨て山（リサイクル） |
| 設計変更 | 1 | 汚染ストックから2枚を永久除外 | → ゲームから除外 |

対応カードストックが尽きたら捨て山をシャッフルして新ストックにする。
手札上限: 3枚。

---

### ラウンドの流れ

```
[準備フェーズ]
  R1: 汚染なし
  R2以降:
    投入枚数 = ceil(前ラウンドの確定利益 / 2)
    キャップ: 最大3枚/ラウンド
    汚染ストックの上から取って出荷山札に混ぜる
    出荷山札をシャッフル

[出荷フェーズ] ← メインのゲームプレイ
  ループ:
    1. 山札から1枚めくる
    2. カード種別で分岐:
       - 製品カード → 手元に確保（利益候補に加算）
       - 不具合カード → 不具合Pt累積。対応カードを使うか判断
       - イベントカード → 即座に効果発動
    3. 累積不具合Pt >= パニック閾値(通常3) → パニック発生！
    4. パニックでなければ「もう1枚めくる」か「止める」を選択

[結果処理]
  パニック時:
    - そのラウンドの利益 → 全損失
    - 汚染ストックから3枚を即座に山札に追加投入（キャップなし）
    - 対応カードはもらえない
  
  自分で止めた時:
    - 確保した製品カードの合計点 → 利益確定（スコアに加算）
    - 確定利益に応じて対応カード入手:
        0〜2点 → 2枚
        3〜5点 → 1枚
        6点+  → なし

[次のラウンドへ]
```

### ゲーム終了
- 8ラウンド完了
- または出荷山札が尽きた時点

## データモデル（TypeScript型定義）

```typescript
// カードの種類
type CardType = 'product' | 'defect' | 'event';

// 製品カード
type ProductCard = {
  type: 'product';
  name: string;       // '量産品' | '高付加価値品' | '大型案件'
  value: number;       // 1, 2, 3
};

// 不具合カード
type DefectCard = {
  type: 'defect';
  severity: 'yellow' | 'red' | 'black';
  name: string;        // '軽微な不具合' | '重大な不具合' | 'リコール級'
  defectPoints: number; // 1, 2, 3
};

// イベントカード
type EventType = 'sns_fire' | 'deadline_pressure' | 'veteran_retire' 
  | 'kaizen' | 'iso_audit' | 'rookie';
type EventCard = {
  type: 'event';
  eventType: EventType;
  name: string;
  description: string;
};

type Card = ProductCard | DefectCard | EventCard;

// 対応カード
type ResponseType = 'first_aid' | 'root_cause' | 'inspection' | 'design_change';
type ResponseCard = {
  responseType: ResponseType;
  name: string;
  description: string;
};

// ゲーム状態
type GamePhase = 'setup' | 'prepare' | 'shipping' | 'result' | 'game_over';

type GameState = {
  phase: GamePhase;
  round: number;              // 1〜8
  maxRounds: number;          // 8（ソロ）
  score: number;              // 累計スコア
  targetScore: number;        // 目標スコア
  difficulty: string;         // 難易度名
  
  // 山札
  drawPile: Card[];           // 出荷山札
  contaminationStock: DefectCard[];  // 汚染ストック
  
  // 対応カード
  responseStock: ResponseCard[];     // 対応カード山
  responseDiscard: ResponseCard[];   // 対応カード捨て山
  responseHand: ResponseCard[];      // 手札（上限3）
  
  // 現在のラウンド状態
  currentRoundProfit: number;        // このラウンドの利益候補
  currentRoundProducts: ProductCard[]; // めくった製品カード
  currentDefectPoints: number;       // 累積不具合Pt
  panicThreshold: number;            // パニック閾値（通常3）
  
  // イベント効果フラグ
  snsFireActive: boolean;            // 次の不具合Pt2倍
  forcedDraws: number;               // 強制めくり残り枚数
  rookieNextRound: boolean;          // 次ラウンド1枚目プレビュー
  canPreviewFirstCard: boolean;      // 今ラウンドプレビュー有効
  waterInspectionActive: boolean;    // 水際検査有効（次の不具合自動無効化）
  
  // 履歴
  roundHistory: RoundResult[];
  lastDrawnCard: Card | null;
  
  // ゲーム終了
  gameResult: 'playing' | 'clear' | 'failed' | 'game_over';
};

type RoundResult = {
  round: number;
  profit: number;
  panicked: boolean;
  cardsDrawn: number;
  defectRate: number;        // そのラウンド開始時の不具合率
};
```

## UI構成

### 画面レイアウト

```
┌─────────────────────────────────────────┐
│  品証パニック  R3/8   スコア: 12/20     │  ← ヘッダー
├─────────────────────────────────────────┤
│                                         │
│  ┌─────────┐   めくったカード:          │
│  │  山札    │   [量産品+1] [高付加+2]   │  ← メインエリア
│  │  残り32  │   [軽微不具合!]           │
│  │  不良18% │                           │
│  └─────────┘   累計利益: 3点            │
│                 不具合Pt: 1/3           │
│                                         │
│  ┌───────────────────────────────────┐  │
│  │ [もう1枚めくる]   [ここで止める]  │  │  ← アクションボタン
│  └───────────────────────────────────┘  │
│                                         │
│  手札: [応急処置] [原因調査]            │  ← 対応カード手札
│                                         │
├─────────────────────────────────────────┤
│  R1: 3点  R2: パニック!  R3: ...        │  ← ラウンド履歴
└─────────────────────────────────────────┘
```

### UI要件
- めくったカードはアニメーション付きで表示（カードフリップ）
- パニック発生時は画面を赤く点滅させる演出
- 不具合率はプログレスバーで視覚化（緑→黄→赤のグラデーション）
- 対応カードは手札エリアからドラッグまたはクリックで使用
- ラウンド開始時に汚染投入の演出（ストックから山札にカードが移動するアニメーション）
- イベントカード発動時はモーダルまたはトースト通知

### デザインの方向性
- 工場・製造業の雰囲気（暗めの背景、警告色の黄・赤、金属質感）
- カードは製品＝青系、不具合＝黄/赤/黒、イベント＝紫系、対応＝緑系で色分け
- フォントは日本語対応のゴシック体

## 実装の優先順位

### Phase 1: コアメカニクス（まずこれだけで動かす）
1. カード生成と山札シャッフル
2. めくり → 製品/不具合の判定
3. パニック判定
4. 止める → 利益確定 + 対応カード入手
5. ラウンド間の汚染投入
6. 8ラウンドでゲーム終了 + スコア表示

### Phase 2: 対応カード
1. 手札管理（上限3枚）
2. 各対応カードの効果実装
3. リサイクル処理
4. 設計変更の永久除外処理

### Phase 3: イベントカード
1. 各イベント効果の実装
2. SNS炎上（不具合Pt2倍）
3. 納期プレッシャー（強制めくり）
4. ベテラン退職（閾値低下）
5. 改善提案（不具合無効化）
6. ISO監査（ボーナス判定）
7. 新人配属（プレビュー）

### Phase 4: 演出・UX
1. カードフリップアニメーション
2. パニック演出
3. 汚染投入アニメーション
4. サウンドエフェクト（任意）
5. ゲーム終了画面（スコアサマリー、ラウンドごとの振り返り）

## テストプレイで確認したいこと

プロトタイプ完成後、以下の観点でプレイして記録する:

1. **体感の面白さ**: めくりのドキドキ感はあるか
2. **汚染カーブ**: 中盤から「怖くなってきた」感があるか
3. **対応カードの価値**: 取りに行く判断が意味を持つか
4. **イベントの影響**: 予測不能な展開が生まれるか
5. **目標スコア**: 「普通」の20点が適切な難易度か
6. **プレイ時間**: 8ラウンドが長すぎないか
7. **設計変更カード**: 使いどころの判断が楽しいか

## 注意事項

- ゲームロジック（gameEngine.ts）はUI非依存の純粋関数で実装する。後で2人対戦やオンラインに拡張しやすくするため
- ランダム処理（シャッフル）はFisher-Yatesアルゴリズムを使用
- テストプレイ中にバランス調整できるよう、constants.ts の値を変えるだけでパラメータ変更できる設計にする
